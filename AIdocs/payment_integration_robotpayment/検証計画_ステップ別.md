# 検証計画（ステップ別・ログ確認付き）

段階的に動作を確かめながら進めるため、検証をステップごとに区切り、各ステップでログを確認できるようにする。  
トラブル時はログを取得し、**原因を明確に追求してから**次に進む。

---

## ログ確認ルール（必須）

- **トラブル発生時**: 必ず `contract_payment` ログを取得する（`php artisan robotpayment:show-test-log`）。
- **原因追求**: ログの時系列・該当ステップのログメッセージを確認し、エラー内容・`response_body`・前後の処理を照合してから原因を特定する。
- **AI・担当者**: 問題報告を受けた際は、**必ずログ内容を確認したうえで**原因分析・修正案を出すこと。ログを見ずに推測のみで結論しない。
- 本ルールは AIdocs に記載済みであり、検証・障害対応時に従う。

（ログ出力先・取得方法は [テストモード接続検証手順.md](テストモード接続検証手順.md) の「テスト結果のログ取得」を参照。）

---

## 検証の区切り方（フェーズ）

| フェーズ | ステップ | 検証後にやること |
|----------|----------|------------------|
| **Phase 1** | Step 1〜2 | ログで「申込フォーム表示」「確認画面表示（POST）」「確認送信→決済ページへリダイレクト」を確認してから次へ |
| **Phase 2** | Step 3〜4 | ログで「決済ページ表示」とブラウザのトークン作成結果を確認。NG ならログ＋Network を確認 |
| **Phase 3** | Step 5〜6 | ログで「決済実行リクエスト受付」「ROBOT PAYMENT gateway_token 送信」（response_body 含む）を確認 |
| **Phase 4** | Step 7〜8 | ログで「決済実行成功」と通知受信（任意）を確認 |

---

## Step 1: 申込フォーム表示

| 項目 | 内容 |
|------|------|
| **操作** | 申込URL（例: http://localhost:8080/billing/）を開く |
| **確認** | プラン一覧・必須項目が表示され、送信できること |
| **ログで確認する項目** | `申込フォーム表示` が contract_payment に出力されていること。plan_count 等が想定どおりか |
| **トラブル時** | ログに「申込フォーム表示」が出ているか。出ていなければルート・ContractController@create・DB（contract_plans）を確認 |

---

## Step 2: 確認送信 → 決済ページへリダイレクト

| 項目 | 内容 |
|------|------|
| **操作** | 必須項目を入力し「確認する」を送信（POST /contract/confirm） |
| **確認** | 確認画面を経ずに **決済ページ**（/contract/payment）へ 302 リダイレクトすること |
| **ログで確認する項目** | `確認画面表示（POST）` の直後に `確認送信→決済ページへリダイレクト`（contract_plan_id 付き）が出力されていること |
| **トラブル時** | リダイレクトしない場合、ログで「確認送信→決済ページへリダイレクト」が出ているか。出ていなければ config('robotpayment.enabled') と store_id、セッションを確認 |

---

## Step 3: 決済ページの表示内容

| 項目 | 内容 |
|------|------|
| **操作** | 決済ページが表示された状態で止め、画面とログを確認する |
| **確認** | 金額サマリー（ta）・カード入力欄・「支払いを実行する」が表示されていること。金額が選択プランと一致すること |
| **ログで確認する項目** | `決済ページ表示` が出力され、contract_plan_id, pattern, am, tx, sf, ta が想定どおりか（買い切り単品なら pattern=one_time_only, am=1 等） |
| **トラブル時** | ログの am/tx/sf/ta と画面表示が一致するか。不一致なら PurchasePatternService・payment ビューを確認 |

---

## Step 4: トークン作成（CPToken.TokenCreate）

| 項目 | 内容 |
|------|------|
| **操作** | テストカード（4444333322221111, CVV 123, 有効期限当月以降, 名義任意）を入力し「支払いを実行する」をクリック |
| **確認** | 「トークン作成に失敗しました」が出ず、3DS 処理へ進むこと（ブラウザ側のため、サーバログにはトークン作成の成否は出ない） |
| **ログで確認する項目** | 「支払いを実行する」クリック時に **カード入力情報（検証用）**（card_length, last4, expiry_mm, expiry_yy）が出力される。テストカードなら card_length=16, last4=1111。トークン作成失敗時は **トークン作成失敗（クライアント）**（err_msg）が続く。成功時はその後に「決済実行リクエスト受付」が出る。ブラウザ Console / Network で RP へのリクエスト成否も確認可 |
| **トラブル時** | リファラーエラー → 決済データ送信元URL。カード番号無効 → ログの「カード入力情報（検証用）」で last4 が 1111 か・card_length が 16 かを確認。トークン失敗時は「トークン作成失敗（クライアント）」の err_msg を確認 |

---

## Step 5: 3Dセキュア認証 → フォーム送信

| 項目 | 内容 |
|------|------|
| **操作** | トークン取得成功後、3DS が実行され、問題なければフォームが POST される（自動で進む） |
| **確認** | 「3Dセキュア認証に失敗しました」が出ず、POST /contract/payment/execute が発行されること |
| **ログで確認する項目** | 次の Step 6 で「決済実行リクエスト受付」が出れば、Step 5 まで正常（ブラウザ→サーバの POST が届いている） |
| **トラブル時** | 3DS 失敗時はブラウザ Console。ログでは Step 6 の「決済実行リクエスト受付」の has_tkn / has_session_data を確認 |

---

## Step 6: 決済実行（サーバ → gateway_token.aspx）

| 項目 | 内容 |
|------|------|
| **操作** | フォーム送信により POST /contract/payment/execute が実行される |
| **確認** | 完了画面へリダイレクトされること。エラー時は決済ページに payment_error とともに戻ること |
| **ログで確認する項目** | `決済実行リクエスト受付`（has_tkn=true, has_session_data=true）→ `ROBOT PAYMENT gateway_token 送信`（http_status, **response_body**）→ `決済実行成功` または `決済実行失敗`（error 内容）。response_body に ER003 がないこと、成功時はカンマ区切り2番目が 1 であること |
| **トラブル時** | response_body を必ず確認。ER003→送信元IP。その他エラーコード→RP 仕様と照合。ログの response_body を貼り付けて原因追求する |

---

## Step 7: 完了画面への遷移

| 項目 | 内容 |
|------|------|
| **操作** | 決済成功後、署名付き URL で完了画面が表示されることを確認する |
| **確認** | URL が /contract/complete/{id}?signature=... であり、完了画面が表示されること |
| **ログで確認する項目** | 直前の `決済実行成功`（contract_id 付き）が出力されていること |
| **トラブル時** | ログで「決済実行成功」が出ているか。出ているのに画面がおかしい場合は URL 生成・ルートを確認 |

---

## Step 8: 初回決済結果通知（任意）

| 項目 | 内容 |
|------|------|
| **操作** | 決済実行後、RP が決済結果通知URLに GET キックバックする（ローカルは ngrok 等が必要） |
| **確認** | 当社エンドポイントが 200 を返し、payment_events に rp_initial_kickback が保存されること |
| **ログで確認する項目** | 通知受信時は RobotPaymentNotifyService 内のログ。gid なし・対応する Payment なしなどの warning が出ていないか |
| **トラブル時** | 通知が来ない場合は RP 管理画面の通知URL・ngrok の URL を確認。ログに「RP 初回通知」系が出ているか確認 |

---

## トラブル時のログ取得と確認の流れ

1. 現象を再現する（または再現した直後の状態を維持する）。
2. `docker compose exec app php artisan robotpayment:show-test-log --lines=300` を実行し、出力を保存または貼り付け用に用意する。
3. **AI・担当者**: 上記ログの時系列を追い、該当フェーズのログメッセージ（申込フォーム表示→確認→決済ページ表示→決済実行リクエスト→gateway_token 送信→成功/失敗）を確認する。
4. response_body や error メッセージを根拠に原因を特定し、修正案を出す。ログを確認せずに推測のみで結論しない。
