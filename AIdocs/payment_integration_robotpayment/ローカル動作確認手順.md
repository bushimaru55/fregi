# ROBOT PAYMENT ローカル動作確認手順

ローカル環境で申込〜決済ページまでの流れを確認するための手順です。

---

## 1. 前提

- PHP 8.1 以上、Composer が利用できること
- `.env` が存在し、DB 接続が有効であること（`.env` が無い場合は `cp .env.example .env` のあと `php artisan key:generate` と DB 設定を編集）

---

## 2. 一括セットアップ（推奨）

次の **1 コマンド** でマイグレーションとテスト用プラン投入まで実行できます。

```bash
cd app
php artisan robotpayment:setup-local
```

実行内容:

- マイグレーション実行（RP 用カラム追加を含む）
- 契約プランが 0 件の場合のみ、テスト用プラン（一回限り・月額）を自動作成
- 次のステップ（.env 設定・起動方法）を表示

---

## 3. 手動で行う場合（マイグレーションのみ）

```bash
cd app
php artisan migrate --force
```

以下が適用されます（未適用の場合）。

- `payments` に RP 用カラム（provider, payment_kind, merchant_order_no, rp_gid, rp_acid, amount_initial, amount_recurring, paid_at）
- `payment_events` に raw_query, rp_gid, rp_acid
- `contract_items` に billing_type

契約プランが無い場合は、別途 `php artisan test:create-plans` でテスト用プランを作成してください。

---

## 4. 環境変数（.env）

**デモアカウントでローカルから開発テストする場合**は、[デモアカウント_ローカル開発テスト.md](デモアカウント_ローカル開発テスト.md) を参照し、RP デモ管理画面で確認した店舗IDを `ROBOTPAYMENT_STORE_ID` に設定してください。

### 4.1 決済を「無効」のまま確認する場合（従来フロー）

```env
ROBOTPAYMENT_ENABLED=false
```

- 申込フォーム → 確認 → **「申し込む」で従来どおり契約・明細のみ作成 → 完了**
- 決済ページは表示されません。

### 4.2 決済ページまで表示する場合

```env
ROBOTPAYMENT_ENABLED=true
ROBOTPAYMENT_STORE_ID=133732
```

- `ROBOTPAYMENT_STORE_ID` は RP の店舗ID（当社: 133732）。テスト環境で別IDの場合は差し替えてください。未設定だと「決済は現在ご利用いただけません」で申込フォームへ戻ります。
- 申込フォーム → 確認 → **決済ページへリダイレクト** → カード入力欄と金額が表示されます。
- 実際のトークン取得・3DS・gateway 送信は RP の環境と店舗IDが有効な場合のみ成功します。ローカルでは「画面遷移と金額表示」の確認が中心になります。

設定反映のため、必要に応じて:

```bash
php artisan config:clear
```

---

## 5. アプリ起動

```bash
cd app
php artisan serve
```

ブラウザで `http://127.0.0.1:8000` を開きます（サブディレクトリで運用している場合はそのURL）。

---

## 6. 確認したい流れ

### パターンA: 決済なし（ROBOTPAYMENT_ENABLED=false）

1. トップ（申込フォーム）でプラン・必須項目を入力
2. 「確認する」→ 確認画面
3. 「申し込む」→ 契約・明細作成 → 完了画面
4. 管理画面の申し込み一覧に 1 件追加されていること

### パターンB: 決済ページまで（ROBOTPAYMENT_ENABLED=true + STORE_ID 設定）

1. トップでプラン・必須項目を入力
2. 「確認する」→ **決済ページへリダイレクト**
3. 請求金額（初回・月額の有無）が表示されていること
4. カード入力欄が表示されていること
5. 「戻る」で確認画面相当に戻れること（セッションがあれば）

### パターンC: 通知の冪等（API のみ）

- 初回通知:  
  `GET http://127.0.0.1:8000/api/robotpayment/notify-initial?gid=1&rst=1&cod=999&acid=1&am=1000&tx=0&sf=0&ta=1000`
- 同じ URL を 2 回呼んでも 2 回とも `200` かつ body が返ること（冪等）
- 自動課金通知:  
  `GET http://127.0.0.1:8000/api/robotpayment/notify-recurring?gid=2&rst=1&cod=999&acid=1&am=1000&tx=0&sf=0&ta=1000`
- 同様に 2 回呼んでも 200 で冪等であること

※ `cod=999` はローカルに存在する契約ID（または payment の merchant_order_no）に合わせると、該当 Payment が更新されます。

## 7. トラブル時

- **「決済は現在ご利用いただけません」**  
  `ROBOTPAYMENT_ENABLED=true` かつ `ROBOTPAYMENT_STORE_ID` が空でないか確認。

- **決済ページで「トークンが取得できませんでした」**  
  RP の JS（CPToken）が読み込まれているか、ブラウザの開発者ツールでエラーを確認。テスト用店舗IDが無効だとトークン取得に失敗します。

- **「店舗設定からリファラーURLを設定してください」とアラートが出る**  
  **決済システムコントロールパネル**（credit.j-payment.co.jp）の「設定」→「決済システム」→「PC用決済フォーム設定」→「決済データ送信元URL」に、ローカル環境のURL（例: `http://localhost:8080`）を登録する。詳細は [デモアカウント_ローカル開発テスト.md](デモアカウント_ローカル開発テスト.md) の「2.4 決済データ送信元URL（リファラー）」を参照。

- **マイグレーションエラー**  
  `php artisan migrate:status` で状態を確認。既存の FREGI 用カラム名（orderid, receiptno 等）と衝突していないか確認。

- **契約ステータスがない**  
  `contract_statuses` に `applied` 等が入っているか確認。必要なら `php artisan db:seed` または該当 Seeder を実行。
