# テストモード 1ステップずつ接続検証手順

ROBOT PAYMENT の「決済システム設定」画面を前提に、テストモードで申込〜決済〜通知を1ステップずつ動作検証する手順です。  
計画: `.cursor/plans/テストモード接続検証計画_ba25d0a2.plan.md` に基づく。

---

## 前提（2種類の管理画面）

- **決済システムコントロールパネル**（credit.j-payment.co.jp）: 決済データ送信元IP・決済データ送信元URL（リファラー）・店舗IDの確認
- **請求管理ロボ**（billing-robo.jp）: 決済結果通知URL・自動課金結果通知URL の登録（[接続に必要な情報.md](接続に必要な情報.md)）

当社アプリのベースURL（例: `http://localhost:8000` または `https://your-app.example.com`）を決め、RP にはそのURLで登録する。

---

## 検証前の準備

### 1. 決済システムCP（決済システム設定）で確認・修正

| 項目 | やること |
|------|----------|
| システムモード | **テストモード**のまま（本番課金されない） |
| 決済データ送信元IP | 当社サーバの送信元IPを登録（例: 203.135.194.26）。`curl -s ifconfig.me` で確認可。 |
| PC用「決済データ送信元URL」 | **当社アプリの決済ページのオリジン**を1件以上登録。例: `http://localhost:8000` または `https://your-app.example.com`（[デモアカウント_ローカル開発テスト.md](デモアカウント_ローカル開発テスト.md) 2.4） |
| PC用「クレジットカード決済完了後転送先URL」 | 当社はサーバ側で署名付き完了URLへリダイレクトするため、空または当社完了ページのURLでよい。 |

※ 「店舗設定からリファラーURLを設定してください」が出る場合は、**決済データ送信元URL** にブラウザで開いているオリジンが登録されているか確認する。

### 2. 請求管理ロボで通知URLを登録

当社アプリが受け取る2本のURLを登録する。

- **決済結果通知URL（初回）**: `https://あなたのドメイン/api/robotpayment/notify-initial`
- **自動課金結果通知URL**: `https://あなたのドメイン/api/robotpayment/notify-recurring`

ローカル検証時は ngrok 等で公開URLを用意するか、「決済実行まで」を先に検証し、通知はステージングで後から検証する（[デモアカウント_ローカル開発テスト.md](デモアカウント_ローカル開発テスト.md) 2.2）。

### 3. .env と config の確認

- `ROBOTPAYMENT_ENABLED=true`
- `ROBOTPAYMENT_STORE_ID=`（決済システムCPの店舗ID。例: 133884）
- `ROBOTPAYMENT_NOTIFY_INITIAL_URL` / `ROBOTPAYMENT_NOTIFY_RECURRING_URL`: 請求管理ロボに登録したURLと一致させる（参照用）
- 反映: `php artisan config:clear`

設定の確認は次で行える。

```bash
php artisan robotpayment:verify-config
```

### 4. テストカード情報（RP カスタマーセンター案内）

テストモードでトークン作成〜決済まで検証する際は、以下を使用する。

| 項目 | 値 |
|------|-----|
| カード番号 | 4444333322221111 |
| CVV | 123 |
| 有効期限 | 当月以降（例: 12/30） |
| 名義（姓・名） | 上記以外は任意の値を入力 |

※ カード番号・CVV・有効期限以外は任意でよい（RP 案内より）。

## 1ステップずつの動作検証

段階的に進める場合は [検証計画_ステップ別.md](検証計画_ステップ別.md) を参照し、フェーズごとにログを確認してから次へ進む。

各ステップで期待結果を確認し、NG の場合は `storage/logs/contract_payment.log` とブラウザの開発者ツール（Console / Network）を確認する。

### Step 1: 申込フォーム表示

- **操作**: 当社アプリの申込URL（`/` または `/contract/create`）を開く。
- **確認**: プラン一覧・必須項目が表示され、送信できること。
- **NG時**: ルート・ContractController@create・DB（contract_plans）を確認。

### Step 2: 確認送信 → 決済ページへリダイレクト

- **操作**: 必須項目を入力し「確認する」を送信（POST /contract/confirm）。
- **確認**: ROBOT PAYMENT 有効時は確認画面を経ずに **GET /contract/payment** へ 302 リダイレクトすること。
- **NG時**: `config('robotpayment.enabled')` と `store_id` が有効か、セッションに `contract_confirm_data` が入っているかを確認。

### Step 3: 決済ページの表示内容

- **操作**: 決済ページが表示されている状態で止める。
- **確認**: 金額サマリー（ta）・カード入力欄・「支払いを実行する」ボタンが表示されること。`store_id`（aid）・`am/tx/sf` が正しく渡っていること。
- **NG時**: ContractController@payment・PurchasePatternService・payment.blade.php を確認。

### Step 4: トークン作成（CPToken.TokenCreate）

- **操作**: テストカード（上記「テストカード情報」参照。カード番号 4444333322221111、CVV 123、有効期限は当月以降、名義は任意）を入力し「支払いを実行する」をクリック。
- **確認**: 「トークン作成に失敗しました」が出ず、次の 3DS 処理へ進むこと。ネットワークで RP のトークン用リクエストが成功していること。
- **NG時**: 決済データ送信元URL（リファラー）・CPToken.js 読み込み・store_id（aid）の一致を確認。

### Step 5: 3Dセキュア認証（ThreeDSAdapter.authenticate）

- **操作**: トークン取得成功後、3DS が実行される（テスト環境ではチャレンジがスキップされる場合あり）。
- **確認**: 「3Dセキュア認証に失敗しました」が出ず、フォームが POST されること。※ Success は「API呼び出し成功」であり、最終成否は決済レスポンスで判断する（仕様書 P.16/17）。
- **NG時**: am/tx/sf/em/pn が決済と同一か、EMV3DSAdapter.js の読み込みを確認。

### Step 6: 決済実行（サーバ → gateway_token.aspx）

- **操作**: フォーム送信（POST /contract/payment/execute、tkn + セッション）。
- **確認**: ER003 が出ないこと（送信元IPが RP に登録済みであること）。サーバログで「ROBOT PAYMENT gateway_token 送信」が出て、gateway の応答が返っていること。テストモードではカード会社には送られず、RP 側のテスト結果で成功/失敗が返る。
- **NG時**: ER003 → 固定IP登録。その他 → ログのレスポンス・RobotPaymentService のパラメータを確認。

### Step 7: 完了画面への遷移

- **操作**: 決済実行が成功した場合のリダイレクト先を確認。
- **確認**: 署名付き URL（`/contract/complete/{id}?signature=...`）へ 302 し、完了画面が表示されること。セッションの `contract_confirm_data` が削除されていること。
- **NG時**: RobotPaymentController@execute の成功時分岐・URL::temporarySignedRoute を確認。

### Step 8: 初回決済結果通知の受信（任意・推奨）

- **操作**: 決済実行後、RP が「決済結果通知URL」に GET キックバックする。
- **確認**: 当社の `GET /api/robotpayment/notify-initial?...` が呼ばれ、200 かつ ContentLength 0 以上（例: "OK"）を返すこと。`payment_events` に `rp_initial_kickback` が保存され、`payments` の status 等が更新されること。
- **ローカル**: RP がローカルURLを叩けないため、ngrok で公開URLを用意するか、通知はステージングで検証する。

## テスト結果のログ取得

問題があった際の修正の参考とするため、テスト結果をログで取得する。

### ログ確認のルール（AIdocs 記載・必須）

- **トラブル発生時**: 必ず contract_payment ログを取得する（下記コマンド）。取得したログを原因追求・報告に用いる。
- **原因追求**: [検証計画_ステップ別.md](検証計画_ステップ別.md) の「ログ確認ルール」および各ステップの「ログで確認する項目」に従い、ログの時系列と該当メッセージを確認してから原因を特定する。
- **AI・担当者**: 問題報告を受けた際は、**必ずログ内容を確認したうえで**原因分析・修正案を出すこと。ログを見ずに推測のみで結論しない。本ルールは AIdocs に記載済みであり、検証・障害対応時に従う。

### ログの出力先

- **チャンネル**: `contract_payment`
- **ファイル**: `storage/logs/contract-payment-YYYY-MM-DD.log`（daily ローテーション、30日保持）

### 記録される主な内容

- 決済ページ表示（contract_plan_id, pattern, am, tx, sf, ta）
- 決済実行リクエスト受付（has_tkn, has_session_data, contract_plan_id）
- ROBOT PAYMENT gateway_token 送信（contract_id, cod, pattern, http_status, **response_body**）
- 決済実行成功 / 決済実行失敗
- 各種エラー（例外メッセージ・トレース）

### 直近ログを取得するコマンド

```bash
# Docker の場合
docker compose exec app php artisan robotpayment:show-test-log

# 直近 300 行を出力する場合
docker compose exec app php artisan robotpayment:show-test-log --lines=300
```

出力をそのまま貼り付けると、問題修正の参考にできる。
