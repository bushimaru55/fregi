# 現在生じている不具合・取りまとめ

請求管理ロボ連携・ROBOT PAYMENT 決済フローで把握している不具合と対策状況です。検証・本番運用時の参照用にまとめています。

---

## 1. 未解消／要検証

### 1.1 API 3 エラー 1340（請求先部署に決済手段が存在しない）

| 項目 | 内容 |
|------|------|
| **現象** | 決済（gateway）は成功し API 1・API 2 も成功するが、**API 3 請求情報登録**でエラー 1340 が返り失敗する。`billing_robo_demands` に請求情報が保存されず、**API 4 請求書発行**も実行されない。 |
| **エラー内容** | `error_code: 1340`, `error_message: "Not Exist 'individual_payment_method_code'."`（請求先部署にデフォルト決済手段が存在しない、または指定した決済情報が紐付いていない）。 |
| **発生条件** | 決済成功後の API 3 呼び出し時。契約 12・13 で再現。 |
| **原因** | 請求管理ロボ側で、請求先部署（individual）にデフォルトの決済手段が紐付いていない。API 3 の demand で `payment_method_code` を送ると「そのコードが部署に存在しない」、送らないと「部署にデフォルト決済手段がない」となり 1340 になる。 |
| **実施した対策** | (1) **API 3**: demand から `payment_method_code` を送らないように変更（デフォルト決済手段を使用する前提）。 (2) **API 1**: 初回成功後、返却された決済情報コードを使って **API 1 を更新で再呼び出し**し、請求先部署に `individual.payment_method_code` を設定して紐付ける。 |
| **状態** | 対策は実装済み。**アプリ再起動後の決済実行で解消しているかは要検証**。再起動せずに決済した場合は古いコードが動き 1340 が継続する。 |
| **確認方法** | アプリ再起動 → 新規申込で決済実行 → ログに「請求管理ロボ API 3 請求情報登録完了」が出るか、`billing_robo_demands` にレコードが入るかを確認。失敗時は「請求先部署への決済手段紐付け（API 1 更新）」の debug ログ有無を確認。 |
| **参照** | [11_phase3_verification_steps.md](11_phase3_verification_steps.md) Step B-8／トラブルシュート、[05_api_03_demand_bulk_upsert.md](05_api_03_demand_bulk_upsert.md) |

## 2. 解消済み（参考）

以下は過去に発生し、設定変更またはコード修正で解消した事象です。

| 事象 | 原因 | 解消方法 |
|------|------|----------|
| **リファラーエラー**（トークン作成失敗） | 決済ページのオリジンが、決済システムCPの「決済データ送信元URL」と一致していない。**メインURL**が優先チェックされ、「複数決済データ送信元URL」のみの登録では通らない。 | 決済システムCP（credit.j-payment.co.jp）の **決済データ送信元URL（メイン）** を、実際に表示するブラウザのオリジン（例: ローカルは `http://localhost:8080`、本番は `https://billing-robo.jp/`）に設定。保存を忘れず実行。 |
| **ER003**（送信元IPの認証に失敗しました） | 決済ゲートウェイ側の「決済データ送信元IP」に、アプリを実行しているサーバ（または Docker）の送信元IPv4 が登録されていない。 | 決済システムCP「決済ゲートウェイ＆CTI決済設定」で、送信元IP（例: `docker compose exec app curl -s -4 ifconfig.me` で確認）を登録。 |

## 3. 注意・フォロー

| 項目 | 内容 |
|------|------|
| **API 1 決済手段紐付けの失敗** | API 1 の「請求先部署への決済手段紐付け」再呼び出しが 400 等で失敗した場合、`contract_payment` には **debug** レベルでのみ記録される。API 3 はそのまま実行されるため、1340 が再発する可能性がある。本番ではログレベル（`config('logging.channels.contract_payment.level')`）や監視で debug も確認できるようにするか、紐付け失敗時に warning に上げる検討を推奨。 |
| **API 4 の 3301 等** | API 3 が成功しても、API 4 請求書発行で「発行日がまだ来ていません」（3301）等が出る場合がある。請求管理ロボの仕様（発行日・送付日・決済期限の条件）に合わせて demand の日付を調整する必要がある。 |
