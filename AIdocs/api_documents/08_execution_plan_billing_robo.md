# 請求管理ロボ連携 実行計画

本ドキュメントは、請求管理ロボ（決済会社独自システム）とのAPI連携について、**方針**と**実行計画**をまとめたものです。

---

## 1. 前提

- **請求管理ロボ**: 決済会社の独自システム。本システムはAPIを通じて請求先登録・請求情報登録・請求書発行・即時決済などの処理を依頼する。
- **本システム**: 契約・プラン・商品のビジネスルールを持ち、「何を・いつ・いくら請求するか」を決定し、請求管理ロボへ送信する。
- **商品**: 初回のみ課金＋月額課金など、複合課金パターンが存在する。

---

## 2. 方針

### 2.1 役割分担

| 役割 | 本システム | 請求管理ロボ |
|------|------------|--------------|
| 商品・料金の定義 | ✅ **正（マスタ）** | 使う場合は「写し」のみ |
| 初回＋月額などの課金ルール | ✅ ここで定義・計算 | 受け取った請求情報をそのまま実行 |
| 請求情報の「内容」の決定 | ✅ 何を・いつ・いくら送るかを決定 | — |
| 請求情報の送信 | APIで送る | 受け取って請求書発行・決済を実行 |

### 2.2 商品マスタ

- **商品の登録・更新は本システムの管理画面のみで行う**。ここを正とする。
- 請求管理ロボ側での商品マスタ登録・更新は**行わない**（二重マスタにしない）。
- 請求情報登録（API 3）では、**item_code を使わず**、本システムの商品・契約から `goods_name`, `price`, `quantity`, `tax_category`, `tax` 等を組み立てて送る。
- ※将来的に請求管理ロボの商品マスタを「省略用」として使う場合は、**本システム → 請求管理ロボの片方向同期**に限定し、請求管理ロボ画面では商品を編集しない運用とする。

### 2.3 複合課金（初回＋月額など）

- 「初回のみ課金＋月額課金」などは、**本システムで「何をいつ送るか」を分解**する。
- 請求管理ロボには、請求情報を**複数行（または複数件）**で送る形で対応する。
  - 例: 初回請求時 → 単発（type=0）で初回金額の1行 ＋ 定期定額（type=1）で月額・開始日・繰り返しの1行。2回目以降は月額分のみ。
- この分解ロジックは本システムの契約・商品・課金ルールに実装する。

---

## 3. 範囲（API）

本システムが利用する請求管理ロボAPIは、[02_scope_billing_robo.md](02_scope_billing_robo.md) のとおり次の5つとする。

| # | 処理 | Path |
|---|------|------|
| 1 | 請求先登録更新 | `POST /api/v1.0/billing/bulk_upsert` |
| 2 | クレジットカード登録（トークン方式 3Dセキュア利用） | `POST /api/v1.0/billing_payment_method/credit_card_token` |
| 3 | 請求情報登録更新 | `POST /api/v1.0/demand/bulk_upsert` |
| 4 | 請求書発行 | `POST /api/v1.0/demand/bulk_issue_bill_select` |
| 5 | 即時決済 | `POST /api/demand/bulk_register` |

- 商品登録更新API（`/api/v1.0/goods/bulk_upsert2` 等）は**本計画では利用しない**。商品マスタは本システムのみで管理する。
