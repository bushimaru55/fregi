# 請求管理ロボ連携 実装順序案

本ドキュメントは [08_execution_plan_billing_robo.md](08_execution_plan_billing_robo.md) の実行計画に基づき、**実装の順番**を提案したものです。依存関係を満たしつつ、小さく検証しながら進められるようにしています。

---

## 前提：完了済み

- 請求管理ロボの接続設定（`config/billing_robo.php`、`.env` の `BILLING_ROBO_*`）
- 疎通テストコマンド（`php artisan billing-robo:ping`）
- DB：契約の `billing_code` / `billing_individual_number` / `billing_individual_code`、決済の `billing_payment_method_number` / `billing_payment_method_code`、`billing_robo_demands` テーブル
- モデル：`Contract`・`Payment` の請求管理ロボ用カラム、`BillingRoboDemand`

## 実装順序案（フェーズ単位）

### フェーズ 1：基盤（API クライアント・共通処理）

| 順 | 内容 | 成果物・参照 |
|----|------|--------------|
| 1.1 | 請求管理ロボ API 共通クライアントの実装（POST・JSON・認証付与・タイムアウト・共通エラーパース） | `App\Services\BillingRobo\BillingRoboApiClient` 等 |
| 1.2 | 共通エラーコードのマッピングと例外／ログ方針の整理（401/400/429/503 等） | 例外クラス or レスポンス DTO、00_billing_robo_api_overview.md |

**完了目安**: 任意の 1 API を共通クライアント経由で呼び出し、成功・エラー両方を扱える状態。

---

### フェーズ 2：商品登録（本システム）

商品マスタは本システムを正とするため、請求先登録・請求情報登録の前に**商品が登録済み**である必要があります。商品登録が済んでいなければ先に進めません。

| 順 | 内容 | 成果物・参照 |
|----|------|--------------|
| 2.1 | 本システムの**商品マスタ**で、商品の登録・更新・参照が管理画面からできる状態にする | 既存の商品マスタ・管理画面（08 の 2.2） |
| 2.2 | 契約プラン・契約明細で参照する商品（商品コード・名称・単価・税区分・課金タイプ等）が一貫して利用できることを確認 | 商品・契約プラン・ContractItem の紐付け |
| 2.3 | 請求管理ロボ連携で必要な項目（goods_name, price, quantity, tax 等を送るため）が商品マスタから取得できるようにする | API 3 の demand 組み立てに利用（05） |

**完了目安**: 管理画面で商品を登録・編集でき、契約・請求情報登録（API 3）で参照する準備ができていること。

---

### フェーズ 3：請求先・決済手段（API 1・2）

| 順 | 内容 | 成果物・参照 |
|----|------|--------------|
| 3.1 | **API 1 請求先登録更新**のリクエスト組み立て・呼び出し・レスポンスから `billing.code` / `individual[].number|code` / `payment[].number|code|cod` を取得して DB に保存 | 03_api_01_billing_bulk_upsert.md |
| 3.2 | 契約登録・更新フローから API 1 を呼ぶタイミングの組み込み（「申込内容確定」時など） | ContractController または専用サービス |
| 3.3 | **API 2 クレジットカード登録**のリクエスト組み立て・呼び出し（billing_code / 決済情報番号 or コード / token） | 04_api_02_credit_card_token.md |
| 3.4 | フロント：トークン取得（CPToken）→ サーバーへ token 送信 → API 2 呼び出しの一連フロー | 既存決済画面との接続、01 シーケンス |

**完了目安**: 契約から請求先・部署・決済情報が API 1 で登録され、クレジット利用時は API 2 でカード登録まで動作すること。

### フェーズ 4：請求情報（API 3）

| 順 | 内容 | 成果物・参照 |
|----|------|--------------|
| 4.1 | 本システムの商品・契約明細から **demand 配列**を組み立てるロジック（item_code は使わず goods_name, price, quantity, tax 等で送信） | 05_api_03_demand_bulk_upsert.md、08 の 2.2 |
| 4.2 | 複合課金（初回＋月額）の分解：初回は単発＋定期定額の複数行、2 回目以降は月額分のみなど | 同上 |

| 4.4 | 契約・請求サイクルに応じて「いつ API 3 を叩くか」のトリガー（申込確定時・月次バッチ等）の整理 | 業務フローに依存 |

**完了目安**: 契約・商品に基づいて demand が組み立てられ、API 3 で登録され、請求情報番号/コードが DB に保持されること。

## 依存関係のイメージ

```
[1] 基盤（API クライアント）
     ↓
[2] 商品登録（本システムの商品マスタ）  ← ここが済んでいなければ先に進まない
     ↓
[3] API 1・2（請求先・カード登録）  ← 契約フローと密結合
     ↓
[4] API 3（請求情報）              ← 商品・契約明細・課金ルールに依存
     ↓
[5] API 4（請求書発行）            ← API 3 の請求情報番号/コードが必要
     ↓
[6] API 5（即時決済）              ← クレジット時のみ。API 3 の内容を別形式で送る
     ↓
[7] フロー統合・エラー整理
     ↓
[8] 結合テスト・検証
```

---

## 進め方のコツ

- **フェーズ 1 完了後**に、API 1 を共通クライアントで呼ぶ実装から入ると、以降の API が同じ土台で書けます。
- **フェーズ 2（商品登録）**を先に済ませておくことで、請求先登録・請求情報登録の実装時に「商品が登録済み」の前提で進められます。商品登録が未完了のまま先に進まないようにします。
- **API 2（カード登録）**はフロント（トークン取得）とサーバー（API 2 呼び出し）の両方が必要なので、3.3 と 3.4 は並行 or 連続で実装するとよいです。
- **API 3**の「demand 組み立て」は、フェーズ 2 で整えた商品マスタ・契約明細・課金ルールに依存します。4.1・4.2 を先に固めることを推奨します。
- **API 4 と API 5**は、どちらを先に実装するかは「まず請求書発行をしたいか、即時決済を優先したいか」で決めて問題ありません。本案では 5 → 6 の順にしています。
